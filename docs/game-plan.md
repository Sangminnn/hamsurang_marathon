# 함수랑 마라톤 게임 기획 초안

## 1. 한 줄 정의

여러 명이 같은 방에 들어와 제한 시간 동안 빠르게 입력해서 전진 거리를 누적하고, 순위 보상으로 재화를 얻어 캐릭터를 꾸민 뒤 다시 경쟁하는 실시간 캐주얼 레이싱 게임.

## 2. 핵심 판단

- 이 게임의 본질은 `정교한 물리 시뮬레이션`이 아니라 `낮은 진입장벽의 실시간 경쟁`이다.
- 따라서 중요한 것은 화려한 액션보다 `입력 반응성`, `순위 표시의 즉시성`, `보상 루프`, `관전자도 이해 가능한 진행 상황`이다.
- Phaser는 **적합하다**. 이유는 2D 연출, 애니메이션, 카메라, UI 레이어, 모바일 대응을 빠르게 만들 수 있기 때문이다.
- 다만 실시간 경쟁의 권위(authority)는 Phaser가 아니라 **서버**가 가져야 한다.
- 즉, `Phaser + WebSocket 기반 authoritative server` 조합이 현재 요구사항에 가장 현실적이다.

## 3. 게임 코어 루프

1. 플레이어가 캐릭터를 선택한다.
2. 매치 룸에 입장한다.
3. 카운트다운 후 레이스 시작.
4. 플레이어는 정해진 입력 액션으로 전진 게이지를 누적한다.
5. 서버가 입력 유효성을 계산해 위치를 갱신한다.
6. 레이스 종료 후 순위와 보상을 지급한다.
7. 보상으로 코스튬, 이펙트, 감정표현, 트랙 연출 등을 해금한다.
8. 꾸민 캐릭터로 다시 입장한다.

이 루프가 유지되려면 `한 판 30~90초`, `결과 확인 10초 내`, `재매칭 5초 내`가 적절하다.

## 4. 추천 게임 모드

### A. 스프린트 레이스

- 4~8인 실시간 방
- 30초 동안 누적 이동 거리 경쟁
- 가장 구현이 쉽고, 랭킹과 보상 설계가 단순함

### B. 체크포인트 레이스

- 짧은 코스에 3개의 구간 보너스 배치
- 특정 구간마다 추가 속도 버프 또는 방해물
- 단순 클릭만으로 지루해지는 문제를 완화

### C. 팀 릴레이 이벤트

- 2:2 또는 3:3
- 각 팀 전체 이동 거리 합산
- 커뮤니티성, 스트리밍 적합성 강화

초기 MVP는 `A 단일 모드`만 추천한다.

## 5. 입력 방식 제안

단순 연타만 두면 매크로와 피로도가 문제다. 아래 중 하나로 설계하는 것이 낫다.

### 권장안: 좌우 번갈아 탭

- `좌/우 버튼` 또는 `A/L`, `←/→` 번갈아 입력
- 올바른 리듬 입력 시 전진
- 같은 버튼 연타는 효율 감소

장점:

- 매크로 난이도 상승
- 리듬감 생김
- 모바일/PC 양쪽 대응 쉬움

### 대안: 타이밍 존 입력

- 이동 게이지가 왕복하고, 초록 구간에서 눌러야 최대 가속
- 연타보다 숙련도를 만들기 좋음

단점:

- 실시간 멀티에서 시인성 설계가 조금 더 복잡함

## 6. 캐릭터 방향성

## 수랑이

- 둥글고 귀여운 실루엣
- 감정 표현이 강점
- 광범위한 코스튬 소화 가능
- 대중성, 굿즈성, 컬렉션 확장성이 높음

## 거북이

- 달리기와 대비되는 캐릭터성 자체가 밈 포인트
- 껍데기, 표정, 장비 변주가 쉬움
- 경쟁심과 우스꽝스러움이 함께 살아남

## 추천 포지셔닝

- 수랑이: `스피드형`, `사랑스러운 반응`, `입문자 선호`

능력치 차등은 초기에 넣지 않는 것이 좋다. 밸런스 리스크만 커지고 커스터마이징 가치가 왜곡된다.

## 8. 보상 및 경제 구조

## 화폐 종류

- `코인`: 기본 플레이 보상
- `젬`: 시즌 보상, 업적, 이벤트 보상

초기에는 화폐를 2개 이상 늘리지 않는 것이 좋다.

## 보상 구조

- 1등: 코인 대량 + 낮은 확률로 희귀 아이템
- 2~3등: 코인 중량
- 4등 이하: 최소 보장 코인
- 연속 플레이 보너스: 3판, 5판, 10판 누적 보상
- 일일 미션: 1등 1회, 레이스 5회, 거북이로 완주 등

## 아이템 희귀도

- Common
- Rare
- Epic
- Legendary

주의:

- 성능 아이템 판매는 초기에 금지
- 이동속도, 입력 판정, 보상 획득량에 직접 영향 주는 과금은 피하는 것이 맞다
- 이 게임은 경쟁 신뢰가 핵심이라 pay-to-win 인식이 생기면 오래 못 간다

## 9. 실시간 경쟁 설계

## 권위 구조

- 클라이언트는 `입력 이벤트`만 전송
- 서버는 `입력 간격`, `유효한 입력 패턴`, `최종 이동 거리`를 계산
- 클라이언트는 서버 상태를 보간해서 표시

이 구조가 필요한 이유:

- 연타 조작은 치트 유인이 높음
- 클라이언트 계산을 믿으면 매크로, 속도 변조, 패킷 조작 방어가 약함

## 방 구조

- 관리자 또는 방장이 대기방 생성
- 빠른 입장 또는 초대 코드 입장
- 최소 2인, 최대 8인 권장
- 인원이 부족하면 봇 또는 카운트다운 후 강제 시작 여부 선택 가능

## 방 운영 역할

- `관리자`: 이벤트성 방 생성, 모드 선택, 공개/비공개 설정, 강제 시작/종료 가능
- `방장`: 일반 유저 기준 방 생성자, 게임 선택, 최대 인원 설정, 준비 상태 확인 가능
- `참가자`: 방 입장, 캐릭터 선택, 준비 완료, 방 이탈 가능

초기 MVP에서는 `관리자`와 `방장`을 분리하지 말고, `방장 권한` 하나로 합치는 것이 맞다.

이유:

- 운영 권한 모델이 단순해짐
- 관리자 전용 기능은 나중에 웹 어드민으로 분리 가능
- MVP 단계에서 권한 구조를 세분화해도 게임 재미 검증에는 도움이 적음

## 예외 처리 규칙

- 카운트다운 중 누군가 `준비 취소`하면 즉시 카운트다운 중단
- 카운트다운 중 누군가 `방 이탈`하면 즉시 카운트다운 중단
- 방장은 인원이 충분할 때 `강제 시작` 가능하게 할 수 있지만, 초기에는 빼는 편이 안전함
- 경기 중 이탈자는 `DNF` 처리하고 결과 테이블에는 남김
- 네트워크 재접속은 레이스 중 복귀 허용보다 `관전자 재진입`부터 지원하는 것이 구현상 안전함

## 모바일 퍼스트 요구사항

- 세로 화면 기준으로 먼저 설계
- 핵심 입력 버튼은 양 엄지로 누르기 쉬운 하단 고정
- 준비 상태, 방 코드, 참가 인원, 카운트다운은 상단 고정
- 레이스 중 장식 UI를 줄이고, 순위/내 위치/남은 시간만 크게 노출
- 한 손 조작도 가능한 대체 입력 배치 검토

초기 기준 레이아웃 권장:

- 상단: 방 정보, 인원 수, 준비 상태, 타이머
- 중앙: 트랙과 캐릭터 진행 상황
- 하단: 좌우 탭 입력 버튼

모바일 퍼스트로 봐야 하는 이유:

- 이 게임은 짧은 세션, 반복 플레이, 즉시 반응이 핵심이라 PC보다 모바일 친화성이 높음
- PC 중심 UI를 먼저 만들면 실제 주 사용 시나리오에서 버튼 크기와 시야 구성이 다시 깨질 가능성이 높음

## 10. 기술 스택 제안

## 프론트엔드

- `Next.js` 또는 `Vite + React`
- 게임 화면은 `Phaser`
- 로비/상점/인벤토리/UI는 `React`
- Phaser를 React 내부 캔버스 영역으로 마운트

### 왜 Phaser인가

- 2D 캐주얼 게임 제작 속도가 빠름
- 스프라이트, 애니메이션, 파티클, 카메라 연출이 충분함
- WebGL/Canvas fallback이 안정적
- 마라톤 레이싱 같은 단순 횡스크롤 표현에 과하지 않음

### Phaser의 한계

- 대규모 MMO급 네트워크 엔진은 아님
- 실시간 동기화, 룸 관리, 치트 방지까지 해결해주지 않음
- UI 전체를 Phaser로 밀어붙이면 상점/인벤토리 유지보수가 오히려 나빠짐

결론:

- `게임 화면만 Phaser`
- `서비스 UI는 React`

## 서버

### 추천안: Node.js + Colyseus 또는 Socket.IO

#### 1순위: Colyseus

- 룸 개념이 명확함
- authoritative multiplayer 패턴에 맞음
- 게임 상태 동기화 구조가 Socket.IO보다 정돈돼 있음
- 소규모 실시간 게임 MVP에 적합

단점:

- 일반 웹 개발자에게는 낯설 수 있음
- 생태계와 레퍼런스는 Socket.IO보다 좁음

#### 2순위: Socket.IO

- 진입장벽이 낮음
- 디버깅과 운영 인력이 구하기 쉬움
- 빠르게 MVP 만들기 좋음

단점:

- 룸 상태, 재접속, authoritative loop를 직접 잘 짜야 함
- 구조를 대충 잡으면 나중에 서버 코드가 쉽게 엉킴

### Colyseus와 Socket.IO의 실제 차이

#### Socket.IO는 무엇인가

- 웹소켓 통신을 쉽게 다루게 해주는 실시간 통신 라이브러리
- 메시지 주고받기, 룸 분리, 재연결 같은 기본 기능 제공
- 하지만 `게임 룸 상태를 어떻게 관리할지`는 대부분 직접 설계해야 함

즉, Socket.IO는 `통신 도구`에 가깝다.

#### Colyseus는 무엇인가

- 멀티플레이 게임 서버 프레임워크
- 룸, 상태 동기화, 메시지 처리, 게임 루프 구조를 게임 기준으로 제공
- authoritative room 모델을 전제로 설계되어 있음

즉, Colyseus는 `게임 서버 뼈대`에 가깝다.

#### 같은 요구사항을 구현할 때 차이

예를 들어 아래 기능이 필요하다고 하자.

- 방 생성
- 참가자 입장/퇴장
- 준비 완료 체크
- 전원 준비 시 카운트다운
- 경기 상태 동기화

Socket.IO에서는:

- 룸 데이터 구조를 직접 만들고
- 준비 여부 저장 로직을 직접 만들고
- 게임 상태 브로드캐스트 규칙도 직접 만들고
- 재접속 또는 중복 입장 처리도 직접 정리해야 한다

Colyseus에서는:

- 룸 클래스 안에서 상태와 메시지를 상대적으로 일관되게 관리할 수 있고
- 상태 스냅샷/패치 동기화 패턴이 이미 정리돼 있어
- `대기방 -> 카운트다운 -> 경기 -> 종료` 같은 흐름을 한 곳에 모으기 쉽다

#### 냉정한 비교

- 팀이 웹 개발은 익숙하지만 게임 서버는 약하다면, `Socket.IO가 시작은 쉬워 보인다`
- 그런데 방 상태와 경기 상태가 섞이기 시작하면, 오히려 `직접 짠 코드의 복잡도`가 빨리 커진다
- Colyseus는 첫 진입이 조금 낯설지만, 게임 방 중심 구조에는 더 잘 맞는다

#### 이 프로젝트에서의 선택 기준

- `빠르게 한번 붙여보고 싶다`면 Socket.IO
- `처음부터 방/상태/레이스 흐름을 안정적으로 관리하고 싶다`면 Colyseus

현재 요구사항이 이미 아래를 포함한다.

- 방 생성
- 게임 선택
- 참가자 출입
- 준비 완료 상태 관리
- 전원 준비 시 카운트다운 시작
- 경기 상태 동기화

이 정도면 단순 채팅방이 아니라 `게임 룸 상태 머신`에 가깝다.

그래서 팀 숙련도가 조금 낮더라도, 장기적으로는 Colyseus가 더 안전한 선택일 가능성이 높다.

### 냉정한 추천

- 팀에 실시간 게임 서버 경험이 적다면 `Socket.IO`로 시작하는 선택도 가능하다.
- 다만 지금 요구사항은 이미 `룸 상태 머신`이 필요하므로, 중간 규모 이상으로 갈수록 Socket.IO 직접 구현 비용이 커질 가능성이 높다.
- 개인적으로는 이 프로젝트 조건에서 `Phaser + React + Colyseus + PostgreSQL` 조합을 추천한다.
- 단, 정말 MVP 속도가 최우선이면 `Socket.IO로 1차 프로토타입`, 검증 후 `Colyseus 전환`도 가능하다. 다만 이 경우 서버를 두 번 설계할 가능성을 감수해야 한다.

## 데이터 저장소

- `PostgreSQL`: 유저, 인벤토리, 재화, 경기 결과
- `Redis`: 룸 캐시, 매치메이킹 큐, 일시 상태

초기 MVP는 Redis 없이도 가능하지만, 실시간 룸이 늘면 붙이는 편이 좋다.

## 인증

- 간편 로그인 또는 익명 로그인 후 계정 승격
- 초기에 회원가입 장벽을 낮추는 것이 중요

## 11. 안티치트 최소 설계

- 초당 허용 입력 수 상한
- 입력 패턴 반복성 검사
- 비정상적으로 일정한 주기 탐지
- 서버 시간 기준 판정
- 경기 로그 저장 및 재검증

완벽한 안티치트는 어렵다. 대신 `이상치 감지 + 빠른 제재 + 서버 권위 계산`이 현실적이다.

## 12. MVP 범위

2~4주 안에 검증 가능한 최소 범위는 아래다.

- 캐릭터 2종: 수랑이, 거북이
- 각 캐릭터 코스튬 5종
- 4인 실시간 방
- 단일 모드: 30초 스프린트
- 입력 방식 1종: 좌우 번갈아 탭
- 모바일 세로 화면 우선 UI
- 방 생성/입장/퇴장/준비 완료/전원 준비 시 카운트다운
- 결과 화면 및 순위 보상
- 코인 상점
- 단순 인벤토리/장착

이 이상을 초반에 넣으면 제작 기간보다 운영 복잡도가 빨리 커질 가능성이 높다.

## 13. 개발 우선순위

### Phase 1. 플레이어블 프로토타입

- 단일 화면에서 클릭/탭으로 전진
- 캐릭터 애니메이션 반응
- 1인 레이스 감각 검증

### Phase 2. 로컬 멀티 시뮬레이션

- 가짜 상대 3명
- 순위 UI와 완주 연출
- 레이스 템포 검증

### Phase 3. 온라인 멀티 MVP

- 룸 생성/입장
- 서버 authoritative position sync
- 결과 보상 지급

### Phase 4. 메타 시스템

- 인벤토리
- 상점
- 장착 프리셋
- 일일 미션

### Phase 5. 라이브 운영 준비

- 시즌 패스 또는 이벤트
- 로그 분석
- 비정상 플레이 탐지

## 14. 제품 리스크

### 리스크 1. 클릭 피로도

- 너무 빠른 연타는 손이 아프고, 이탈이 빠르다
- 해결: 번갈아 탭, 짧은 경기, 리듬성 입력

### 리스크 2. 매크로

- 입력 게임은 매크로 유혹이 강하다
- 해결: 서버 판정, 패턴 검출, 반복 간격 제한

### 리스크 3. 보상 인플레이션

- 재화를 쉽게 뿌리면 상점 동기가 빨리 소진된다
- 해결: 시즌 한정, 파츠 컬렉션, 점진적 해금

### 리스크 4. 실시간성 과대설계

- 이 게임은 FPS가 아니므로 초저지연 인프라까지는 필요 없다
- 100~200ms 수준의 지연은 보간으로 충분히 흡수 가능하다
- 따라서 WebRTC data channel까지 바로 갈 필요는 거의 없다

## 15. 최종 제안

- 게임 장르는 `실시간 캐주얼 경쟁 레이싱`으로 명확히 정의한다.
- 초기 MVP는 `수랑이/거북이 2캐릭터`, `4인 방`, `30초 스프린트`, `좌우 번갈아 탭`, `코인 보상`, `코스튬 장착`까지만 만든다.
- 클라이언트 게임 렌더링은 `Phaser`가 적합하다.
- 서비스 UI는 `React`로 분리한다.
- 실시간 서버는 `Colyseus` 우선 검토, 팀 숙련도상 부담이 크면 `Socket.IO`로 시작한다.
- WebRTC는 현재 요구사항에서는 과하다.
- 가장 먼저 검증할 것은 `재미`가 아니라 정확히는 `30초 안에 경쟁심이 생기는가`다.
